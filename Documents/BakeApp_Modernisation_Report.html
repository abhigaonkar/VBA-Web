Problem Being Solved
Operators manually entering bake completions into SAP is error-prone and time-consuming. This tool automates SAP postings (COR6 confirmations, MIGO goods movements, MB52/MB51 stock queries).
Tracks which unbaked tubes went into which oven, for how long, yielding which baked items — providing full lot traceability from unbaked batch → pallet card → baked item.
Manages sand (quartz) feedstock transfers between warehouse and furnaces with FIFO ordering using SAP MB51/MB52 transaction data.
Supports ● Online (live SAP GUI scripting) and ● Offline (local-only logging) modes for resilience when SAP is unavailable.
User Personas
Furnace Operator — enters bake records, selects hamper/strapping containers, triggers SAP saves. (e.g. wickhamc, Belcherc, richcreekr)
Shift Supervisor — monitors SAP status, reviews "Entered in SAP?" column, clears discrepancies via ketchup replay routine.
Sand Handler — uses SandForm to transfer quartz sand batches from warehouse (0015/0100) to furnace locations (NF21–NF38) with FIFO batch ordering.
SAP Transactions Automated
COR6 — Process Order Confirmation MIGO — Goods Movement (261/309/311) MB52 — Warehouse Stock by Material MB51 — Material Document List (FIFO) MM03 — Material Master Display COR3 — Process Order Display ZMMS2S — Custom Sand Transfer COP1 — Handling Unit Create COWBHU — HU Goods Movement
Key Inputs
Pallet card number, oven selection (NV0A–NV0J), bake hours (soak time only)
Baked process order, baked/unbaked item numbers, batch numbers (up to 16)
Baked qty, unbaked qty, hamper selection, strapping, comments
Sand item/batch, transfer quantity, source/destination furnace location
Key Outputs
SAP COR6 confirmation (step 20 + step 40) with yield, scrap, hours, operator
MIGO 261/309 goods movements for material issue and hamper/strapping
Pallet labels printed to shop-floor printers (HEBPRNP054–057)
OvenLog.xlsx remote backup on SharePoint UNC path + TransactionLog.csv audit trail

Business Rules (inferred from VBA + formulas)
IDRule
BR-01Time per tube = Total bake hours ÷ unbaked qty. Stored in Sheet1 col K. Used for bake efficiency metrics.
BR-02Sand yield rates per oven type stored in Sheet2 (kg/inch). Tolerance check: (target − actual) / actual. Variants for 130", 154", 138", 112" tube sizes.
BR-03FIFO sand consumption: when multiple batches exist at a furnace, MB51 document numbers determine the oldest batch (lowest MBLNR = consumed first).
BR-04Process orders validated against SAP COR3 — order must not be "complete" before confirming. If complete, operator is prompted to confirm overwrite.
BR-05Scrap = unbaked qty − baked qty (if positive). Scrap qty and reason code (from 93-code defect list) are required in SAP COR6 AFRUD-GRUND field.
BR-06Hamper item numbers resolved by size from reference table: 52"→134978, 58"→134977, 66"→134976, 70"→134975, 86"→134974, 110"→134908 (+ multi-pack variants).
BR-07HU batch format: {year-digit}{month-letter}{furnace-code}{start}-{end}. Month = A(Jan)…L(Dec). e.g. 5MT001-004 = 2025, Month=May, Furnace=T, tubes 001-004.
BR-08Source stock defaults to location 0020 (shop floor). Hamper storage defaults to 0100. Constant 147664 used for strapping MIGO line items.

Non-Functional Requirements & Risks
Performance expectations, security gaps, data integrity patterns, and maintainability risks.
Performance
Ketchup routine loops all offline rows — can be slow on large backlogs. Sheet1 has 1,650+ rows.
SAP GUI scripting adds 10–30 seconds per record depending on SAP response time. COR6 step 20+40 is the heaviest call.
Remote backup to SharePoint UNC adds network delay per save — handled with user-visible "Saving, please wait" message.
Security & Access Control
No in-application authentication. Access control relies entirely on Windows file permissions and SAP user login.
Operator identity captured from Environ("Username") — can be spoofed on a shared PC.
VBA code not password-protected (binary readable). SAP credentials not embedded — uses existing SAP GUI session.
Maintainability Risks
RiskSeverityDetail
Hardcoded file pathsHighBoth local dev paths (C:\Users\PARKINSONJ\OneDrive...) and network paths (\\US-FSP01\...) are hardcoded in Module1 with a dual-path If switch — path changes require VBA edits.
SAP screen path fragilityHighSAP GUI scripting uses hard string IDs (e.g. wnd[0]/usr/tabsTABSTRIP_5115/tabpKOWE/...). Any SAP UI upgrade or customisation change will silently break automation.
Hardcoded employee IDMedEmployee number "97747" hardcoded in clsCOR6 for the AFRUD-PERNR field. Will fail when employee rolls off.
Single-developer originMedCode references developer's local OneDrive path, Debug.Print statements, and TODO comments. Bus factor = 1.
No persistent error loggingMedStdErrorHandler shows MsgBox but does not write errors to a persistent log. Intermittent SAP failures are undetectable after the fact.
SAP Logon startup fragilityLowstartNetweaver uses SendKeys with fixed-position UP arrow presses to navigate SAP Logon list — breaks if list order changes.
Sheet2 column couplingLowHamper item#s, oven codes, defect codes, yield rates, printer names all live in Sheet2 with no named ranges. Column position changes break all VBA references.
Data Integrity Strengths
Qty confirmation dialogue shown before SAP post ("Your unbaked qty is X, baked qty is Y. Is this correct?").
Dual-write: local sheet first, remote backup second, SAP third — ensures local record exists even if network or SAP fails.
Col J SAP flag provides clear reconciliation audit — FALSE rows are visually identifiable for supervisor review.
Data Integrity Gaps
No rollback: if SAP posting succeeds but local write fails, SAP flag stays FALSE — risk of duplicate posting on retry.
Offline records rely on ketchup routine to replay — if workbook is closed without running ketchup, records may never reach SAP.
No transaction locking — two operators saving simultaneously could cause row collision on nextEmptyRow lookup.
